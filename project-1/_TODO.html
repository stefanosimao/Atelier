Goal

The goal of this exercise is to create a responsive music website using HTML5 for the structure and content and CSS for styling.

Your website should have at least the following pages:

  1. a home page
  2. a song catalog page
  3. a music player page
  4. a song upload form page.

To get started, you can use the skeleton files we provide in the gitlab classroom project.

Your markup should be semantically correct: for example, use the <header> tag for the HEADER section; we will not accept vintage solutions which only use <div class="header"> elements.

Both the desktop and mobile layouts should be implemented using CSS only. Use the same HTML markup for both layouts.

Submitting your solution

Your project solution should be commited and pushed to your github class repository. Make sure your name is written in every file you submit. Anonymous solutions will not be graded.

Please sign up for a mastery check slot only after you have pushed your solution.

Task 1. Holy grail layout

Time to get started with the basic structure of your website which will be none other than the classic holy grail layout with a responsive twist.

Content Structure

The holy grail layout consists of a header, a navigation section, a sidebar, the main content and a footer section.

1. Header

The header should display the name of your webpage. Feel free to put a logo if you have one.

2. Navigation

Links to all the pages of your site. Hovering on a link should transition the default background the link to a color of your liking within 250 milliseconds.

3. Main Content

This depends on the page (see below).

4. Sidebar

The sidebar should contain links to your favourite musicians.

5. Footer

Additional information like copyright, the date when the page was last updated and links to legal disclaimers.

Styling and Layout

Your site should use a consistent style shared by all the pages. If you need to customize the layout/formatting of individual pages you can also add CSS rules dedicated to each page.

Use flexbox to implement this layout for all your pages. Pay attention to the design mockups provided.

Mobile

For mobile-size viewports (less than than 768 pixels), your design should look like shown below:

The mobile version should satisfy the following requirements:

  • All sections should fill horizontally the visible viewport.
  • The NAVIGATION and the SIDEBAR are positioned above and below the MAIN CONTENT respectively.

Desktop

For desktop-size viewports (equal or more than 768 pixels), your design should look like shown below:

The desktop version should satisfy the following requirements:

  • The content of the page should horizontally fill the visible viewport.
  • The height of the NAVIGATION and SIDEBAR is expected to stretch all the way to the FOOTER even when there's no content inside them.
  • The FOOTER is supposed be "sticky" to the bottom of the page. This means that even if there's not enough content in the MAIN CONTENT section of your page to push the footer the bottom of the window, the FOOTER should still stick to the bottom. However, if the main content does not fit in the browser viewport, the FOOTER should be pushed out of view (the user will need to scroll to find the footer at the bottom of the page).

Both the desktop and mobile layouts should be implemented using CSS only. Use the same HTML markup for both layouts.

Task 2. Home page

Main Content

  • A list with links to your top 10 favourite songs found in the collection
  • An <audio> element to play the first song mentioned in the list
  • Some statistics about the content of your music collection (for example: how many songs, artists, albums, genres). You can use placeholders for the actual numbers, we will compute the values later on.
  • More links to dive into specific sets of songs (e.g., by the same artist, in the same album, of the same genre).

Styling and Layout

The homepage content should not overflow horizontally. Otherwise you are free to style the main content of the homepage as you prefer.

Task 3. Song Catalog

Look for the songs.html file. Style it with CSS so that it is displayed as a table on desktop and as a set of cards on mobile, without changing the markup.

Styling and Layout

Make sure that the currently playing song is displayed differently than the rest.

Desktop

The table should occupy all available horizontal space. Using CSS, attempt to manually adjust the column width so that the content fits.

The table should look like a table, even if the HTML markup does not use table elements, so experiment with styling borders and background colors to highlight headers.

The action buttons/links to delete, edit, download and mark the song as favourite should be displayed only when the mouse hovers on the corresponding table row.

Task 4. Player page

This page should display an audio player together with information about the song being played.

While the standard HTML audio element can display its own controls, the goal is to create your own player user interface mockup, which we will connect to the audio player in a future assignment using JavaScript.

Styling and Layout

Note that the style for the desktop version of the player page is already provided in the style.css file; you should create the corresponding markup elements without changing the style.

After you have built the HTML markup, then you can create your own styles for the mobile version. Also in this case, the constraint is to make efficient use of the limited space available. For example, the volume control is usually not shown as the users will directly change it using the mobile phone keys.

Task 5. Song Upload Form

Main Content

The page should contain a form with the following input fields:

  • a single-line text field for the song title
  • a single-line text field for the song artist
  • a single-line text field for the song album
  • a drop-down field to select the song genre (with at least 5 different options)
  • a multi-line text field for the optional song description
  • a field to upload the song .mp3 file
  • a field to denote whether the song should be marked as favourite or not
  • a slider to rate the quality of the song (from 0 to 10)
  • the submit button

Every field should have a <label> that denotes what this field expects as a value. Clicking on the label should focus the corresponding form input field. On the Desktop, opt for a two-column layout where on the left you have the labels and on the right the fields. The submit button doesn't need a label since it can display the "Submit" text within its markup.

Pay extra attention to using the correct input type fields, buttons and attributes for the form.

Task 6. Optional Features

  1. Navigation

    Highlight the navigation entry corresponding to the currently active page.

  2. Custom Fonts

    Your site may optionally also load and use at least one custom font. Feel free to either use Web fonts, or to specify one using the @font-face CSS rule.

  3. Form Placeholders

    Before the user enters information the form fields are empty. Add some example text content which will disappear as soon as the user starts typing.

  4. Favourite Star

    Turn the favourite checkbox input into a star (or whatever shape you prefer) which changes color to indicate whether the checkbox input is checked or not. Use only HTML and CSS to implement this.

  5. Interactive Table and Form

    As the user moves the mouse over the table/form elements or enters information in the form, the style of the elements should smoothly change to highlight the current focus of attention.

  6. Scroll Snap

    The mobile version of the song table should use CSS scroll snap so that as the user scrolls through the songs, only one song at a time is displayed on the phone screen.

You should update the README.md file enumerating the optional features you implemented.
